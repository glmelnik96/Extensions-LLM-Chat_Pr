## 0. TL;DR за 30 секунд

- **Цель:** одна кнопка «Синхронизировать» выравнивает по аудио все видео/аудио-дорожки многокамерной съёмки на таймлайне Premiere — не «1 видео + 1 аудио», а **весь таймлайн** (N камер + M рекордеров).
- **Почему отдельный плагин:** чтобы не раздувать LLM-чат-плагин; синк — детерминированный DSP-инструмент без LLM. Общий с родителем — только инфраструктурный слой (ffmpeg, host-мост, CEP-обвязка).
- **Что уже готово в родителе и переиспользуется:** чтение всего таймлайна, извлечение RMS через ffmpeg, ремап media→sequence time, субкадровый сдвиг клипа (`move_clip`/ticks), ripple-сдвиг, паттерн карточек инструментов. ~60% фундамента.
- **Что писать с нуля (ядро):** кросс-корреляция аудио-огибающих → детект офсета; графовая раскладка N×M к опорной дорожке; multi-clip экстрактор.
- **Главный риск:** точность. RMS@50–500мс слишком груб для ±1 кадра. Нужен **спайк** до любой разработки (раздел 9).
- **Стратегический контекст:** дедлайн миграции CEP→UXP ~сентябрь 2026. Любой новый тяжёлый ExtendScript-код таймлайна увеличивает миграционный долг — закладывать тонкий host-слой.

## 1. Что это за плагин и зачем

**«ИИ: синхронизация»** (рабочее имя; bundleId предложить вида `com.gleb.aisync`) — CEP-панель для Premiere Pro 2024+,
которая автоматически синхронизирует по звуку клипы многокамерной/многомикрофонной съёмки.

**Пользовательский сценарий:**
1. Монтажёр накидал на таймлайн дорожки: V1/V2/V3 — камеры, A1/A2 — петлички/рекордеры. Всё рассинхронено (разный старт записи).
2. Открывает панель → «Синхронизировать таймлайн».
3. Плагин извлекает аудио каждой дорожки, считает попарные офсеты к опорной дорожке, показывает карточку-превью (кто на сколько кадров сдвинется).
4. Apply → клипы двигаются `move_clip`-ом с субкадровой точностью. Всё синхронно.

**Чем отличаемся от Syncaila/PluralEyes:**
- **Syncaila** — standalone, XML/AAF round-trip (export → sync → reimport), плодит дубли клипов и теряет метаданные.
- **PluralEyes** — CEP-расширение, но тоже через XML-round-trip внутри Premiere.
- **Мы** — читаем снапшот таймлайна напрямую (`getTimelineSnapshot`), считаем офсеты, двигаем клипы `move_clip`. **Без XML, без реимпорта, без дублей.** Это наше UX-преимущество.

**Out of scope (минимум для MVP):**
- Авто-хронология по датам файлов / именам (Syncaila так раскладывает клипы без аудио-матча) — позже.
- Создание multicam-clip Premiere — не нужно, у нас уже есть audio-driven свитчер в родителе.
- Continuous drift-коррекция (растяжение по таймкоду) — Syncaila этого тоже не делает, ставим один офсет на клип.

## 3. Доменная справка: как работает синк (Syncaila/PluralEyes)

- **Вход:** многодорожечный таймлайн (камеры + рекордеры), без опоры на таймкод.
- **Алгоритм:** сопоставление по **аудио-волне** — кросс-корреляция огибающих/спектра. Доп. сигналы для хронологии клипов без аудио-матча: даты файлов, имена, таймкод.
- **Группировка:** синхронизируют **группами клипов**, а не парами — лишние синхронные клипы усиливают слабые матчи.
- **Точность:** возможна ошибка ±1 кадр при сильном эхо/реверберации. **Один офсет на клип**, без непрерывной drift-коррекции.
- **Параметры (из Syncaila):** Search accuracy (Low…Max), Match threshold (ниже порог → больше клипов, но больше ложных), Audio source (All mixed / First stream / First channel).

→ Подробный разбор и сравнение — в исходном анализе (этот хэндоф — его продолжение). Источники:
`syncaila.com/help`, `provideocoalition.com/sync-shootout-pluraleyes-vs-syncaila`.

```
┌─────────────────────────────────────────────────┐
│  CEP Panel (Chromium + Node.js)  — НОВЫЙ плагин  │
│  client/index.html + panel.js                    │
│                                                  │
│  ┌────────────────────────────────────────────┐ │
│  │ client/shared/ (IIFE, ES5)                 │ │
│  │ • audio-envelope.js  ← ffmpeg → огибающая  │ │  (адаптация audio-preprocess.js родителя)
│  │ • sync-core.js       ← КРОСС-КОРРЕЛЯЦИЯ ★   │ │  (ПИШЕМ С НУЛЯ — ядро)
│  │ • sync-graph.js      ← раскладка N×M, якорь │ │  (ПИШЕМ С НУЛЯ)
│  │ • track-extractor.js ← аудио по клипам      │ │  (расширение rmsExtractor родителя)
│  │ • sync-proposal.js   ← превью офсетов        │ │
│  │ • bridge-premiere.js ← мост (из родителя)   │ │
│  └────────────────────┬───────────────────────┘ │
│                       │ CSInterface.evalScript   │
│  ┌────────────────────▼───────────────────────┐ │
│  │ host/premiere-sync.jsx — ТОНКИЙ host        │ │
│  │ getTimelineSnapshot · moveClip(ticks) ·     │ │  (выжимка из premiere.jsx родителя)
│  │ shiftRipple · createBackupSequence          │ │
│  └────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
         │                        │
   ffmpeg (envelope)        Adobe Premiere Pro (DOM)
```

**Поток данных (L2, «весь таймлайн к опорной»):**
1. `getTimelineSnapshot()` → список всех клипов V/A с `mediaPath`, `inPoint`, `start/end`.
2. `track-extractor` → для каждой дорожки ffmpeg извлекает аудио-огибающую высокого разрешения (5–10 мс), ремап в sequence-time.
3. `sync-core` → попарная кросс-корреляция каждой дорожки с **опорной** (самая длинная/самая громкая) → офсет в секундах (float, субкадр).
4. `sync-graph` → согласование: опорная = 0, остальные = свой офсет; отсев матчей ниже порога корреляции (помечаются «не синхронизировано»).
5. `sync-proposal` → карточка: дорожка → сдвиг (кадры/мс) + уверенность матча.
6. Apply → `createBackupSequence` (чекпоинт, как в родителе) → `moveClip` каждой дорожки на её офсет (ticks).

## 6. Ядро: модуль `sync-core.js` (что именно реализовать)

**Задача:** по двум аудио-огибающим `a[]`, `b[]` (равномерная сетка, шаг `dt`) найти сдвиг `τ`, при котором
они максимально совпадают.

**Базовый алгоритм (L1):**
- Нормализованная кросс-корреляция (zero-mean, unit-variance) по диапазону лагов ±maxLagSec.
- Пик корреляции → `τ`; значение пика → **confidence** (для Match threshold).
- Параболическая интерполяция вокруг пика → субсэмпловая точность (важно для ±1 кадра при грубой сетке).

**Производительность:**
- Прямая корреляция O(N·L) — ок для коротких/L1. Для длинного материала — **FFT-корреляция** (overlap-add):
  нужен FFT на чистом JS (radix-2, ES5). Это основной источник сложности L3.
- Чанкинг: коррелировать на «громких» окнах (где есть речь/хлопок), а не на тишине — быстрее и точнее.

**Точность — ключевое:**
- Кадр при 25fps = 40мс, при 50fps = 20мс. Чтобы попадать в ≤1 кадр, **огибающая ≤10мс + параболическая интерполяция**.
  Если RMS-огибающей не хватает → корреляция по PCM-семплам (тяжелее, но точнее). Решает спайк (раздел 9).

**Тестируемость (pure logic, vm-loader как в родителе):**
- Сгенерировать синтетический сигнал, сдвинуть на известный `τ`, проверить, что `syncCore.findOffset(a,b)` возвращает `τ` в пределах одного шага сетки. Шум/эхо как регрессионные кейсы.

## 7. Дорожная карта (инкрементально, walking skeleton)

| Уровень | Что | Объём/риск | Результат |
|---|---|---|---|
| **L0 — Спайк** | Извлечь огибающие 2 клипов с известным сдвигом, прогнать прототип корреляции, проверить ±1 кадр | XS / решает судьбу | Ответ: хватает ли RMS-огибающей или нужен PCM |
| **L1 — 1↔1** | tool card «Синхронизировать дорожку по опорной» + `sync-core` + `move_clip` | S–M | Выровнять одну дорожку к одной. Минимально полезно |
| **L2 — N→опорная** | Выбор якорной дорожки, попарная корреляция всех к ней, карточка-превью офсетов, чекпоинт, batch-move | M | **Целевой «весь таймлайн»** для типового подкаста/интервью (1 клип на дорожку) |
| **L3 — полный Syncaila** | Multi-clip на дорожку, FFT-ускорение, кэш офсетов, авто-хронология по датам/именам, клипы без аудио | L / высокий | Полноценный sync-движок. Только при бизнес-спросе |

Не доводить L1 до идеала, пока L2 не работает end-to-end (паттерн walking skeleton родителя).

## 8. Ключевые риски и как их снять

| Риск | Суть | Митигейшн |
|---|---|---|
| **Точность vs разрешение огибающей** (главный) | RMS@0.5с не даёт кадровой точности | L0-спайк до разработки; параболическая интерполяция; при провале — PCM-корреляция |
| **Производительность на длинном материале** | Корреляция многочасовой съёмки в CEP-Node | FFT overlap-add; корреляция только по «громким» окнам; чанкинг + abort |
| **ExtendScript-долг перед UXP (сент. 2026)** | Новый тяжёлый host-код таймлайна увеличивает миграцию | Host держать **тонким** (3–4 функции); вся логика — в JS pure-modules, которые переживут миграцию |
| **Дрейф/echo → ошибка кадра** | Reverb ломает корреляцию (как у Syncaila) | Confidence-порог; помечать слабые матчи «проверьте вручную», не двигать молча |
| **Несколько клипов на дорожке** | MVP-экстрактор берёт первый клип | Явно ограничить L1/L2 «1 клип на дорожку» + честный варнинг; multi-clip — L3 |
| **Линкованные A/V пары** | `move_clip` двигает связанные клипы вместе | Учесть линковку (родитель уже это делает в `move_clip`, premiere.jsx:1283) |

## 9. Первый шаг — спайк L0 (делать раньше всего)

Цель: проверить, достижима ли кадровая точность кросс-корреляцией огибающих **до** написания плагина.

1. В **родительском** плагине (он открыт и подключён к Premiere) взять реальный проект с двумя дорожками,
   у которых заведомо разный старт (или искусственно сдвинуть клип на N кадров).
2. Через `node tools/cep-debug.mjs evalfile <probe>` извлечь `mediaPath` обоих клипов из `getTimelineSnapshot`.
3. ffmpeg: вытащить огибающую с окном 5–10мс (`computeRmsTimeline` с малым `windowSec`, или сырой PCM через `astats`/`aresample`).
4. Прогнать прототип нормализованной кросс-корреляции (можно прямо в Node-скрипте `tools/`), сравнить найденный `τ` с известным сдвигом.
5. **Критерий успеха:** ошибка ≤ 1 кадр на чистом звуке; деградация предсказуема на зашумлённом.

Если спайк зелёный — L1 строится поверх него почти напрямую. Если красный — пересмотреть подход (PCM-корреляция / спектральные фичи) до старта.

## 13. Открытые вопросы для уточнения у пользователя

- Целевая точность: достаточно ±1 кадр (как Syncaila) или нужна субкадровая?
- Типовой материал: всегда 1 клип на дорожку (подкаст/интервью) или бывают перезапуски камер (→ сразу L3)?
- Нужен ли визуальный превью-вейвформ в карточке, или достаточно текстового «дорожка → сдвиг N кадров»?
- Объединять с родителем в один установочный bundle или держать полностью отдельным расширением?
