# План монтажа по смыслам (`propose_montage_plan`) — дизайн

Дата: 2026-07-05. Статус: утверждён пользователем (вариант 2 из брейнсторма).

## Цель

Северная звезда продукта: агент **собирает ролики по смыслам**, а не просто владеет
инструментами. Первый сценарий (приоритет A): «сожми этот N-минутный материал до M минут,
сохрани суть». Агент строит структурированный план (блоки keep/cut с темами и причинами),
пользователь читает и правит план ДО деструктива, применение — существующим надёжным путём.

Приоритеты сценариев: A (сокращение) → B (тематическая выжимка, получается бесплатно) →
C (тизер) → D (сборка по сценарию). C/D — вне скоупа v1 (требуют сборки новой секвенции).

## Ключевое решение

Вся арифметика и контроль качества — детерминированный код; за LLM — только семантика
(что оставить, что резать, почему). Новый инструмент — **тонкий слой** над существующей
инфраструктурой `propose_transcript_cuts` (карточка, padding/snap/merge, верификация,
apply с чекпоинтом и `_timelineDiff`), без дублирования.

## 1. Контракт инструмента `propose_montage_plan`

```js
{
  sequenceKey: string,            // обязательный (имя секвенции)
  targetDurationSec: number,      // обязательный, > 0
  blocks: [                       // весь транскрипт, каждый абзац ровно в одном блоке
    { action: 'keep', paragraphs: {from: 0, to: 3},  theme: 'Знакомство и завязка' },
    { action: 'cut',  paragraphs: {from: 4, to: 6},  reason: 'Повтор мысли из блока 1' },
    ...
  ],
  summary: string                 // 1-2 предложения, что произойдёт
}
```

- Гранулярность — **абзац** (v1): резы автоматически по смысловым границам;
  sentence-уровень осознанно не поддержан (host-путь его не умеет, см. P0-C).
- `theme` обязателен для keep-блоков (нужен для карточки), `reason` — для cut-блоков.

## 2. Валидатор плана — `client/shared/montage-plan.js` (чистые функции, без DOM)

`validateMontagePlan(plan, entry)` где `entry` — транскрипт-кэш
(`ContextStore.findTranscriptEntry`), возвращает `{ok, errors[], warnings[], stats}`:

- **Структура**: blocks непустой; action ∈ {keep, cut}; paragraphs.from/to — целые,
  from ≤ to, в диапазоне абзацев entry.
- **Покрытие**: каждый абзац ровно в одном блоке — дыра или перекрытие → error
  с номерами абзацев.
- **Хронометраж**: длительности из paragraphs entry (НЕ из LLM). Сумма keep vs цель:
  промах > ±10% → error с точными цифрами и направлением
  («получилось 21:40 при цели 15:00 — убери ещё ~6:40» / «недобор: … верни ~2:10 или
  предложи пользователю меньшую цель»).
- **Причины**: cut без непустого reason → error; keep без theme → error.
- **Темы/главы**: если в entry есть `topics` — глава, все абзацы которой попали в cut,
  даёт **warning** «тема 'X' пожертвована целиком» (не error: жертва допустима,
  но должна быть явной — попадает в карточку).
- `stats`: `{keepSec, cutSec, targetSec, keepBlocks, cutBlocks, sacrificedTopics[]}`.

Ошибки возвращаются агенту (он перебалансирует план; retry guard страхует от зацикливания).

## 3. Executor `execProposeMontagePlan` (panel.js)

1. Найти entry по sequenceKey (staleness-гейт как в `_resolveRemoveRefs` — stale → error).
2. `validateMontagePlan` → errors → вернуть агенту `{error, _planStats}`.
3. Развернуть cut-блоки в список `removeRefs = [{paragraph: i, reason}]`
   (диапазон from..to → отдельные ref'ы).
4. Делегировать в существующий `execProposeTranscriptCuts({removeRefs, sequenceKey,
   targetDurationSec, summary, removeSummary из cut-блоков, keepSummary из keep-блоков})` —
   переиспользуются padding/snap/merge, финальная верификация, `_pendingProposal`.
5. Прикрепить к `_pendingProposal` поле `planBlocks` (+ `sacrificedTopics`) и перерендерить
   карточку — блоки отображаются секцией плана.
6. Вернуть агенту `{ok, status: 'waiting_user_confirmation', _verification, _planStats}`.

Правки плана пользователем текстом («блок 4 оставь») → агент перевызывает инструмент →
карточка заменяется. Применение — существующий `applyPendingProposal` без изменений
(чекпоинт, `_timelineDiff`, per-interval results).

## 4. Карточка плана (расширение `renderPendingProposalCard`)

При наличии `planBlocks` в proposal (kind остаётся `transcript_cuts`):

- Секция «📋 План монтажа»: блоки по порядку — keep: `✓ [tc–tc] Тема · длительность`,
  cut: `✗ [tc–tc] · длительность · причина`; таймкоды кликабельные (`_tcJumpEl`).
- Итог: «49:40 → 15:12 (цель 15:00 ✓)» — переиспользуется target-badge
  (порог перекраски ±10% для плана).
- Warnings про пожертвованные темы — в блоке warnings карточки.
- Списки keep/remove-интервалов ниже сворачиваются (детализация), план — главное.

## 5. Промпт-протокол

- **Схема инструмента** в TOOLS (panel.js) + короткое упоминание в prompts.js:
  когда запрос «сожми до N минут / собери по смыслу» — использовать
  `propose_montage_plan`, НЕ голый `propose_transcript_cuts`.
- **Builtin-стартер** «Монтаж по смыслам» (категория textmontage,
  conversation-starters.js) с systemPromptAddon-алгоритмом:
  1. Уточнить целевой хронометраж, если не задан.
  2. `get_timeline_snapshot` → sequenceName.
  3. `get_transcript_structure` → абзацы + topics.
  4. Понять материал: темы, драматургия (завязка → развитие → финал).
  5. Построить блоки: у каждого cut причина из закрытого списка
     (повтор / вода / слабый кусок / офтоп), у каждого keep — тема.
  6. `propose_montage_plan`; ошибки валидатора → перебалансировать и перевызвать.
  7. Финальный ответ цитирует цифры из `_planStats`/`_verification`, не свои.
- Требование репозитория (conversation-starters.js:22): стартер добавляется только
  с systemPromptAddon и живым смок-тестом — выполняется (live e2e ниже).

## 6. UI при старте панели

- **Welcome-карточка** (panel.js:3993+): добавить возможность «Сжать ролик до нужной
  длины с сохранением сути — план на подтверждение» с примером запроса.
- **Кнопки снизу**: существующий starters-container — новый builtin-стартер появляется
  там автоматически; отдельных новых кнопок не требуется.

## 7. Тесты и приёмка (критерий B — утверждён)

- **Юнит** (`tests/montage-plan.test.mjs`): валидатор целиком — покрытие
  (дыры/перекрытия/диапазоны), арифметика ±10% в обе стороны (граничные 10.0%),
  причины/темы, warnings пожертвованных topics, разворот блоков в removeRefs,
  stats. Схема инструмента — в существующем стиле tool-тестов.
- **Live e2e** (CDP, метод из памяти): подкаст «Атом» (53 мин) на КОПИИ секвенции
  (backup → рез копии → возврат оригинала) → «сожми до 15 минут» → карточка плана
  читается пользователем → Apply → `_timelineDiff.match === true` → пользователь
  смотрит ролик. Субъективная оценка связности автором — финальный судья.
- Формальные проверки плана (хронометраж ±10%, резы по границам абзацев, причины,
  учёт тем) выполняются валидатором автоматически при каждом вызове.

## Вне скоупа v1 (осознанно)

- Сборка новой секвенции из кусков (сценарии C/D) — отдельный host-инструмент, позже.
- Sentence-уровень гранулярности плана.
- Drag&drop-редактирование плана в карточке (правки — текстом через агента).
