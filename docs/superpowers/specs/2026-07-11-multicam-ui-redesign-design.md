# Авто-MultiCam: редизайн карточки настроек

**Дата:** 2026-07-11
**Статус:** дизайн утверждён, ждёт спеки-ревью → план реализации
**Область:** `client/unified/index2.html` (карточка `#card-multicam`), `client/unified/panel.js` (display-bindings, чтение в dispatch `case 'multicam'`)

## Проблема

В карточке «🎥 Авто-MultiCam (β)» сейчас **16 контролов** лежат плоским списком одного визуального веса: 2 селекта (Дорожки, Пресет), 12 слайдеров, 2 чекбокса. Всё видно сразу и выглядит одинаково важным. Следствия:

- Пресеты (`calm`/`dynamic`) — готовый ответ «не хочу крутить 12 слайдеров» — тонут вторым пунктом, под ними стена, обесценивающая их смысл.
- Новичок не знает, что трогать; опытный не видит, что с чем связано.
- Похожие пары («Мин./Макс. план», «Привязка»/«Сдвиг привязки») разбросаны без визуальной группировки.

Задача — **чисто презентационная**: переставить и сгруппировать контролы, спрятать редкие под сворачиваемый блок, уточнить названия и подсказки. **Логика плана (multicam-plan.js, deterministic-pipelines.js) не трогается.**

## Выбранная модель — «Гибрид: базовые видны + advanced свёрнут»

### Всегда видны (4 контрола)

| Контрол | id | Роль |
|---|---|---|
| Дорожки | `mc-map-mode` (+`mc-mapping`) | структурный: неверный маппинг ломает всё |
| Пресет | `mc-preset` (+`mc-preset-save`) | быстрый старт, теперь реально главный |
| Мин. план | `mc-minhold` | самый интуитивный: «как часто переключать камеры» |
| Порог лидера | `mc-margin` | «насколько уверенно брать план» |

### Свёрнуто под «⚙ Тонкая настройка» — 4 секции (12 контролов)

Механизм: нативный `<details>`/`<summary>` (внешний блок **всегда стартует свёрнутым** — без localStorage). Секции внутри — заголовки (обычные `<div>`-подзаголовки, не вложенные `<details>`, чтобы не плодить клики). Все элементы остаются в DOM → чтение значений и apply-пресета работают без правок JS-логики.

**Секция 1 — Ритм и перебивки:** `mc-maxhold` (Макс. план), `mc-overlap` (Терпеть перебивки), `mc-maxall` (Длина общего плана), `mc-wide-silence` (Общий на паузах), `mc-wide-overlap` (Общий на перебивках)

**Секция 2 — Чувствительность звука:** `mc-silence` (Порог тишины), `mc-smooth` (Сглаживание)

**Секция 3 — Точность резов:** `mc-snap` (Привязка к паузам), `mc-snapoff` (Сдвиг привязки)

**Секция 4 — Вариативность и анализ:** `mc-jitter` (Вариативность), `mc-seed` (Дубль), `mc-framesec` (Шаг анализа)

Логика группировки — по вопросу пользователя «что я хочу поменять»: *слишком часто/редко* → секция 1; *ловит шум / не тот спикер* → топ + секция 2; *резы посреди слова* → секция 3; *слишком ровно / хочу другой прогон* → секция 4.

## Изменения названий (3 лейбла)

| id | Было | Стало | Почему |
|---|---|---|---|
| `mc-overlap` | Анти-перебивка | **Терпеть перебивки** | «анти-X» — неуклюжий новояз; глагол понятнее |
| `mc-maxall` | Длина вставки | **Длина общего плана** | «вставки» чего? — теперь явно |
| `mc-seed` | Вариант | **Дубль** | «дубль» = «дай другой прогон», точнее по смыслу |

Остальные лейблы без изменений. **Ни один `id` не меняется** — panel.js читает по id, переименование сломало бы чтение.

## Подсказки (`title`) — единое правило: что делает + куда крутить

**Всегда видны:**
- **Дорожки** — «Где общий план, какие микрофоны → каким гостям. Авто подходит, если A1→V2, A2→V3… Микрофоны не по порядку — выбери "Вручную".»
- **Пресет** — «Готовый набор: "Спокойный" — редкие плавные переключения (интервью), "Динамичный" — частые (живой подкаст). 💾 сохранит текущие слайдеры.»
- **Мин. план** — «Минимум секунд на одной камере. Влево = чаще режем, живее. Вправо = спокойнее, реже.»
- **Порог лидера** — «Насколько микрофон громче остальных, чтобы забрать план. Вправо = увереннее держим одного, реже дёргаемся на чужой звук.»

**Ритм и перебивки:**
- **Макс. план** — «Дольше этого один спикер в кадре не держится — вставляем общий план, чтобы картинка не застыла.»
- **Терпеть перебивки** — «Короткие реплики поверх (кросс-ток) короче этого НЕ уводят в общий план — держим говорящего. Больше = терпимее. 0 = выкл.»
- **Длина общего плана** — «Сколько длится общий план, вставленный в длинный монолог. Меньше = короткие мелькания, больше = спокойные вставки.»
- **Общий на паузах** — «Пауза (никто не говорит) → общий план. Выкл = держим камеру последнего говорившего.»
- **Общий на перебивках** — «Говорят разом → общий план. Выкл = держим последнего говорившего.»

**Чувствительность звука:**
- **Порог тишины** — «Тише этого микрофон считается молчащим. Вправо = порог ниже (строже, реже слышим тишину).»
- **Сглаживание** — «Гасит покадровое дрожание камеры. Больше = плавнее; слишком много = проспим быстрые реплики.»

**Точность резов:**
- **Привязка к паузам** — «Подтягивает рез к ближайшей паузе / началу слова, чтобы не резать посреди слова. 0 = выкл. Больше = сильнее.»
- **Сдвиг привязки** — «Тонкий сдвиг реза: минус = раньше (до первого слова), плюс = позже. Только при "Привязка" > 0.»

**Вариативность и анализ:**
- **Вариативность** — «Случайный разброс длины планов, чтобы нарезка не была механически ровной. 0 = выкл.»
- **Дубль** — «Перетасовка рисунка разброса. Не понравился прогон → сдвинь, получишь другой. Только при "Вариативность" > 0.»
- **Шаг анализа** — «Шаг анализа звука. Влево = точнее границы резов, но дольше. 50мс — оптимум для речи.»

## Что НЕ меняется

- id всех контролов, диапазоны/шаги/дефолты слайдеров.
- panel.js: логика чтения значений в dispatch, display-bindings (`bindSlider` и IIFE), apply/save пресетов. Порядок чтения id в коде не зависит от порядка в DOM → перестановка HTML безопасна.
- multicam-plan.js, deterministic-pipelines.js — без изменений.
- Кнопка «Проанализировать и предложить нарезку» и `#proposal-multicam`.

## Риски и проверка

- **Риск:** сворачивание убирает контрол из видимости, но чтение по id в dispatch продолжает работать (элемент в DOM). **Проверка:** прогнать multicam через CDP при свёрнутом блоке, убедиться что все 4 tier-параметра доходят до `buildSwitchPlan` (спай-паттерн, как в предыдущей работе).
- **Риск:** пресеты выставляют значения слайдеров, спрятанных в `<details>` — сеттеры по id работают независимо от видимости. **Проверка:** применить пресет при свёрнутом блоке, развернуть, убедиться что слайдеры сдвинулись.
- **Риск:** CSS `<summary>` может конфликтовать со стилями панели (маркер списка, отступы). **Проверка:** визуально в панели — заголовок аккуратный, стрелка тоггла видна.

Автотестов на HTML-разметку в проекте нет; проверка — CDP-прогон в живой панели + визуальный осмотр. Юнит-логика не затронута, сюита 763/763 должна остаться зелёной.
