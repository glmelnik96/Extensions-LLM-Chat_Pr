# Точные резы (тишино-снап) + Рилс: vision по клипам таймлайна

Дата: 2026-07-20. Статус: дизайн утверждён (вариант А по обоим вопросам).

Две независимые доработки одного релиза:

1. Монтаж по смыслам (и все резы по транскрипту) обрезает слова — границы
   резов ставятся на неточные таймкоды Whisper-сегментов без страховки.
2. Рилс позиционирует кадр одним vision-оффсетом на весь файл — а должен
   адаптировать каждый клип таймлайна (спикер в центре композиции).

---

## Часть 1. Точные резы: тишино-снап границ

### Проблема

Границы вырезаемых интервалов = границы Whisper-абзацев/сегментов.
Таймкоды сегментов Cloud.ru Whisper неточны (сдвиг до ~0.3с и больше —
подтверждено на караоке). Снап в `execProposeTranscriptCuts` привязывает
интервалы к этим же неточным границам; padding отсутствует;
`audioAnalysis.silences` в монтажном пути не используется (только в
cut_fillers). Итог: рез попадает в слово.

### Решение (вариант А)

Рез переносится в **центр ближайшей физической тишины** (ffmpeg
silencedetect), fallback — padding наружу от вырезаемого интервала.

**Место:** общий путь `execProposeTranscriptCuts` (panel.js) — после
существующего `snapIntervalsToSegmentBoundaries` и **до**
`mergeRemoveIntervals`. Выигрывают оба потребителя: монтаж по смыслам
(`execProposeMontagePlan` делегирует сюда) и чатовые резы по транскрипту.

**Логика:** новая чистая функция в `client/shared/deterministic-pipelines.js`:

```
DeterministicPipelines.refineCutBoundaries(intervals, silences, opts)
  → { intervals: [...], stats: { snapped, padded, dropped } }
```

Для каждой границы вырезаемого интервала `[t0, t1]`:

1. **Снап к тишине.** Ищем тишину длительностью ≥ `minSilenceSec` (0.2с),
   чей центр в окне ± `windowSec` (0.7с) от границы. Ближайший центр
   побеждает. Граница переносится в центр тишины: рез в физической
   тишине — слово не обрезано независимо от точности Whisper.
2. **Fallback: padding наружу.** Тишины в окне нет (плотная речь) →
   вырезаемый интервал сжимается на `padSec` (0.15с) с этой стороны:
   `t0 += padSec` / `t1 −= padSec`. При ошибке таймкода лучше оставить
   лишний хвост звука, чем отрезать слово из оставляемой речи.
3. **Guards** (в самой функции):
   - после уточнения `t1 − t0 < 0.3с` → интервал отбрасывается
     (dropped, не режем);
   - перенос границы не дальше `windowSec` от входного значения;
   - вход не мутируется; невалидные интервалы (NaN, инверсия)
     отбрасываются;
   - обе формы тишин: `{startSec,endSec}` и `{start,end}` (как в
     рилс-пайплайне).

**После** уточнения — существующий `mergeRemoveIntervals` схлопывает
возможные новые пересечения. Host-валидация пересечений и границ
остаётся last line of defense (не меняется).

**Деградация:** у записи транскрипта нет `audioAnalysis.silences`
(старые записи) → silences = [], работает только padding (шаг 2).
Ничего не падает.

**Отчёт:** в предложении резов панель добавляет строку вида
«Границы: N по тишине, M с отступом, K отброшено (<0.3с)».

### Тесты (Node, tests/)

- граница рядом с тишиной → центр тишины (обе формы полей);
- тишины нет → padding наружу с обеих сторон;
- интервал после уточнения короче 0.3с → dropped;
- тишина за пределами окна ±0.7с → игнорируется, работает padding;
- выбор ближайшей из двух тишин в окне;
- silences = null/[] → чистый padding;
- вход не мутируется; NaN/инверсия → отброс;
- уточнение создало пересечение → merge схлопывает (интеграционно
  с mergeRemoveIntervals).

## Часть 2. Рилс: vision по клипам таймлайна

### Проблема

Vision-оффсет считается один на **файл** (кадр из середины файла,
`vDur/2`) и применяется ко всем клипам этого файла. Если спикер в разных
отрезках стоит в разных местах кадра — все клипы получают один
компромиссный сдвиг. Нужно: кадр из середины **каждого клипа таймлайна**
(с учётом inPoint исходника), оффсет на клип.

### Решение (вариант А: батч по 8 + дедуп)

**Host (premiere.jsx → 2.14.0):** `getVerticalReframeSources` дополняется
полями на клип: `startSec`, `endSec`, `inPointSec` (по образцу
`getFrameSources`, где они уже вычисляются). Аддитивно — существующие
потребители не ломаются.

**Панель (toolsRunReels), заменяет per-file vision-цикл:**

1. Source-время кадра клипа:
   `frameSec = inPointSec + (endSec − startSec) / 2` — середина именно
   того отрезка, что стоит на таймлайне.
2. **Дедуп:** клипы одного `mediaPath`, чьи `frameSec` ближе 2с, делят
   один кадр (типовой случай — один дубль, порезанный монтажом).
3. Кадры: `extractFrameJpeg(mediaPath, frameSec, {maxWidth: 768})` —
   по одному на уникальную группу.
4. **Батч-vision:** до 8 кадров в один запрос (паттерн describe_frames):
   нумерованные изображения + текстовый промпт, ответ
   `[{"i": <номер>, "cx": 0..1}]`. temperature 0,
   `enableThinking: false`, **без** `responseFormat` (MiniMax-M3 на
   Cloud.ru с ним возвращает пустой content — проверено 17.07).
5. Оффсет применяется **per-clip**: результат маппится на
   `trackIndex/clipIndex`. `planVerticalReframe` получает vision-оффсеты
   адресно (trackIndex/clipIndex), а не по имени клипа.
6. Ручной «Фокус камер» (`rl-offsets`) — без изменений: матч по имени,
   побеждает vision (клип, покрытый manual-оффсетом, в vision не идёт).

**Чистая логика** — новые функции в `client/shared/reels-pipeline.js`
(TDD на Node):

```
planClipFrames(clips, {dedupeSec})   → { frames:[{mediaPath, frameSec,
                                          clipRefs:[{trackIndex,clipIndex}]}],
                                          skipped:[...] }
buildVisionBatches(frames, {batchSize}) → [[frameIdx,...], ...]
parseVisionBatchAnswer(text, count)  → [cx|null, ...]
assignClipOffsets(frames, cxList)    → [{trackIndex, clipIndex,
                                          offsetPct}, ...]
```

`planVerticalReframe` (deterministic-pipelines.js): помимо существующего
матча по имени (manual) принимает адресные оффсеты
`{trackIndex, clipIndex, offsetPct}`; приоритет: manual (имя) →
адресный vision → центр 0.5.

**Деградации (не фатальны, сборка продолжается):**

- BRAW / `nest:` — vision пропускается (как сейчас), центр;
- нет `inPointSec` (host вернул null) → fallback на середину файла
  (текущее поведение);
- ошибка извлечения кадра или vision-запроса → центр для этих клипов
  + пометка в отчёте;
- невалидный `cx` в ответе (не число, вне [0,1]) → null → центр.

**Отчёт панели:** вместо «файл.mp4: vision −5%» —
«vision: N клипов, M кадров, K запросов» + перечень деградаций.

### Тесты (Node, tests/reels-pipeline.test.mjs)

- planClipFrames: середина клипа с inPoint; дедуп ближе 2с (один кадр,
  два clipRefs); разные mediaPath не дедупятся; nest/без-пути → skipped;
  inPointSec=null → fallback-маркер;
- buildVisionBatches: 1 кадр → 1 батч; 9 кадров → [8,1]; 0 → [];
- parseVisionBatchAnswer: честный JSON-массив; JSON в markdown-обёртке;
  пропущенные индексы → null; мусор → все null; cx вне [0,1] → null;
- assignClipOffsets: cx→offsetPct((cx−0.5)·100), распределение по всем
  clipRefs группы; null → клип не получает оффсет;
- planVerticalReframe: адресный оффсет побеждает центр; manual по имени
  побеждает адресный; кламп в границы кадра — не сломан (регресс).

## Порядок реализации

Части независимы; в одном плане — сначала Часть 1 (резы, болит на каждом
монтаже), затем Часть 2 (рилс). Live-валидация каждой части по методу
CDP-прогонов на тест-проекте TEst.

## Вне скоупа

- Keyframes у Motion (динамическое слежение за спикером внутри клипа).
- RMS-подстройка границ реза (вариант В — отклонён в пользу А).
- Пословные таймкоды Whisper (Cloud.ru их не отдаёт).
