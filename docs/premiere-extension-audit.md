# Аудит расширения (Premiere Pro, CEP)

Документ фиксирует **что подтверждено в работе**, **что с оговорками**, **что не срабатывает** и **какие правки уже пробовали** — без превращения в список задач на разработку.

**Обновлено:** 2026-04-06 · ориентир хоста `premiere.jsx` ~2.4.0 (маркеры, файловый кэш, `move_clip` / ripple, `shift_timeline_ripple`).

---

## Качество транскрибации и разметки (слабое звено)

Пайплайн **Whisper (FM) → сегменты с таймкодами → кэш → решения LLM (что резать / куда маркеры)** в проде выглядит **слабым по качеству анализа**, не только по интеграции.

| Наблюдение | Смысл |
|------------|--------|
| **Тайминг** | Сегменты и границы фраз **не совпадают** с реальными таймкодами на таймлайне: вырез или маркер попадает **не в тот** отрезок. Источники: погрешность Whisper, склейка чанков In–Out, смещение относительно экспортированного аудио vs позиции клипа на секвенции. |
| **Смысл** | То, что модель **вычленяет** как «блоки», «главы», «лишнее по смыслу», **часто не совпадает** с ожиданием оператора; ошибки накладываются на неточные границы сегментов. |

**Направления ускорения качества (для будущих итераций, не как обещание):** выравнивание/форсированная подгонка границ к ключевым кадрам или к энергии сигнала; другая модель или постобработка сегментов; явная верификация интервалов перед `apply_transcript_cuts`; сжатый human-in-the-loop в UI.

## Скорость работы плагина

Субъективно плагин **ощущается медленным**; ускорение нужно ставить как **явную цель**, а не только «пофиксить баги».

| Узкое место | Комментарий |
|-------------|-------------|
| **Сеть FM** | Каждый шаг агента — запрос `chat/completions`; составной запрос при `maxSteps` даёт серию round-trip. |
| **Транскрибация** | Экспорт In–Out / очередь клипов / чанки, загрузка в Whisper, запись кэша — последовательно и тяжело по диску и API. |
| **ExtendScript** | `evalScript` синхронный; длинные операции на хосте блокируют панель; авто-снимок после правок умножает вызовы. |
| **CEP / Node** | Подготовка файлов, ffmpeg, чтение кэша с диска — без жёсткого бюджета времени в UX. |

**Направления (идеи):** кэшировать снимок и не перезапрашивать без необходимости; сокращать число шагов для типовых команд; параллель только там, где безопасно; индикаторы прогресса и отмена на длинных хост-операциях (где технически возможно); рассмотреть более лёгкую модель или streaming для простых запросов.

## Подтверждено работает

| Область | Что имеется в виду |
|--------|---------------------|
| **Общий кэш транскриптов** | После транскрибации транскрипт **виден и в «ИИ: маркеры по структуре», и в «ИИ: монтаж по тексту»** — раньше ломалось из‑за раздельного `localStorage` у разных Extension Id. Сейчас канон — файл `~/.extensions_llm_chat_pr/_llm_transcript_cache.json` и merge с прочими путями в `client/shared/context-store.js` (корень расширения, `host/`, bundle, при необходимости CEP `userData`). |
| **Транскрибация In–Out (пайплайн)** | Подготовка аудио → FM Whisper → запись сегментов в кэш по **имени секвенции** согласована с промптами (`sequenceKey`). **Точность таймингов и смысловой разметки** — отдельная проблема (см. выше). |

## Работает с оговорками

| Область | Оговорка |
|--------|-----------|
| **Индикатор транскрипта (LED)** | При **переходе между окнами** панелей обновление «есть / нет в кэше» **подключается не сразу** — состояние может отстать до следующего события, которое перечитывает кэш (идея на будущее: обновление по `visibilitychange`/фокусу панели). |

## Не работает или не подтверждено на таймлайне

### Маркеры из чата не оказываются на секвенции (корректно)

- **Симптом:** запросы в «ИИ: маркеры по структуре»; в ответе инструмента успех и разумные `timeSec`, на линейке маркеров нет, один в нуле, или позиции не совпадают.
- **Гипотеза:** `addSequenceMarkers` в `host/premiere.jsx` — как именно в `Sequence.markers.createMarker` передаётся время (объект `Time` / `ticks` vs явные тики от `sequence.timebase`, тип аргумента в вашей версии PP).
- **Что уже пробовали:** перейти на расчёт тиков от timebase и вызов `createMarker` от тиков (версия хоста с этим упором); ответ `created` по-прежнему может отражать **входные** секунды, а не фактические позиции в Premiere.
- **Итог аудита:** для пользователя **маркеры на таймлайне по-прежнему не вызываются / не совпадают** — нужна дальнейшая проверка в целевой сборке PP и уточнение API.

### «Перенеси клип в начало» одним сообщением — не выполняется

- **Симптом:** сообщение ассистента о **лимите шагов (12)** за одно пользовательское сообщение; напоминание разбить задачу; уточнение, что «Стоп» рвёт ожидание FM, но не откатывает уже ушедший ExtendScript.
- **Почему так:** один запрос = один проход `runAgentLoop` с ограничением `maxSteps` (`client/shared/agent-loop.js`). Цепочка «снимок → план → `move_clip` / правки → снова снимок» съедает шаги быстро. Отдельно в хосте у `move_clip` известны случаи, когда Premiere **не сдвигает** клип (коллизии, тихий сбой присвоения `start`/`end`), при этом ответ может оставаться успешным.
- **Что уже пробовали:** промпты про очередь подзадач; для хоста — `move_clip` с ripple-insert по умолчанию, `shift_timeline_ripple`; явное сообщение пользователю при исчерпании шагов.
- **Итог аудита:** формулировка «одним коротким сообщением перенеси в начало» **остаётся ненадёжной** — упирается и в лимит шагов, и в реальное поведение API перемещения.

## Прочие ограничения (кратко; детали в коде/гайдах)

- **Lift** (вырезать интервал без смыкания): долгое время везде доминировал ripple; отдельные ветки lift добавлялись точечно — не все сценарии «дыра на таймлайне» гарантированы одной фразой в чате.
- **`set_clip_enabled` по одному `nodeId`:** после сплитов один исходник даёт несколько кусков с разными границами — отключается не весь «логический» клип без перечисления всех `nodeId` или расширения инструмента.
- **Агент и дубликаты после split:** риск зацикливания и лишних шагов — см. `docs/KNOWN_ISSUES_AND_TEST_GAPS.md`.

## Пустая панель CEP

Если окно панели тёмное без UI: перезагрузка расширения, **Window → Developer Tools** → Console. После внедрения `client/shared/panel-bootstrap.js` (сразу после `__PANEL_ID__` в `index.html`) при падении скрипта сверху может появиться блок с текстом ошибки.
