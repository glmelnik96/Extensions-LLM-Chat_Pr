> (Cloud.ru + локальный host + ffmpeg).

## Где они объективно лучше (и почему)
1. **Vision** — судят монтаж по картинке, не только по тексту. Для нас, чья цель — «монтаж по
   смыслам», это прямой пробел: смысл часто в кадре, а не в словах.
2. **Устойчивость длинных операций** — WS + heartbeat + reconnect. У нас длинные вызовы —
   fire-and-forget через CDP-костыль, в проде — таймауты клиента.
3. **UX ошибок** — доменные коды + человеко-читаемые инструкции. У нас generic.
4. **Ширина фич** — джамп-каты, диаризация, авто-вертикаль, b-roll, цветокор, смарт-субтитры.
5. **Прозрачность стоимости** — контекст-поповер (% окна, system vs actions).

## Где мы лучше (сохранить как moat)
- Приватность (ничего не уходит в облако кроме LLM/Whisper-вызовов на Cloud.ru).
- Мгновенный host (код на диске, не round-trip на каждый tool-call).
- Самовалидация деструктивных host-операций (reject, не clamp) — у них этого нет.
- Заточка под RU-контур (Cloud.ru тарифы, ₽-счётчик).

## Принципы приоритизации
- **Сначала надёжность** (Волна 1): новые фичи на дырявом фундаменте усилят боль, не ценность.
- **Мерить качеством ролика**, не «код работает» — каждую фичу e2e на реальном мультикам-материале
  (1_SYNCED и др.), с ground-truth.
- **Только сведённый вход** для монтажа по смыслам (мультикам → nest/single-cam).
- **Не жертвовать moat**: любую «облачную» фичу конкурента реализуем локально/на Cloud.ru.

## Ближайший шаг (предложение)
Начать с Волны 1, п.1–3 (429 Retry-After + валидатор пересечений + пустые Whisper-segments) —
подтверждены/дёшевы, снимают реальные P0/P1. Перед кодом — грепнуть VERIFY-пункты, написать

## Мультикам: покадровый пинг-понг + анти-дребезг в UI (11.07.2026, live 6_SYNCED 80 мин)
Глубокий live-тест Авто-MultiCam на активной 3-камерной секвенции (тихая запись) вскрыл
критический баг стабильности и UX-разрыв. Ветка `fix/multicam-minhold-cap`.

1. **`enforceMinHold` — покадровый пинг-понг (P0).** Анализ давал **3799 сегментов, 74%
   короче minHold, минимум 0.05с (1 кадр)** — нарезка мигала покадрово (ровно тот AutoPod-
   фейл, который мы декларируем как убитый). Причина: цикл `while (changed && safety++ < 1000)`
   поглощал ровно один коротыш за проход, после 1000 удалений молча обрывался, оставляя
   тысячи под-minHold склеек; рескан-с-нуля после каждого `splice` давал ещё и O(n²)
   (медленный анализ). Юнит-тесты не ловили: на входах < 1000 коротышей потолок не срабатывал.
   **Фикс:** один forward-проход O(n) без потолка — короткий отдаёт время соседу, длинные не
   флипаются. **Live на той же секвенции:** 3799→**1011** сегментов, под-minHold 2802→**0**,
   медиана 0.6с→**5.62с**, минимум 0.05с→**1.5с**. Регресс покрыт тестом на >1000 коротышей.
2. **Анти-дребезг рычаги в UI (UX-паритет).** UI показывал 5 из ~14 параметров плана;
   ключевые рычаги против дребезга были скрыты. Выведены «Сглаживание» (`smoothingWindow`,
   кадры, majority-vote) и «Анти-перебивка» (`overlapWideMinSec`, сек — наш анти-AutoPod-
   differentiator: кратковременный кросс-ток не уводит в общий план). Заодно чинит проброс:
   `multicamFromAudio` форвардил whitelist параметров в `buildSwitchPlan` **без**
   `smoothingWindow` — ползунок молчал бы. Проброс обоих + эффект подтверждены вживую (CDP).

**AutoPod (ресёрч ~15 источников):** единый тариф $29/мес, только громкость (без диаризации),
один мик = одна камера, выход — захардкоженные резы (не живой мультикам). Топ-жалобы —
кросс-ток-хаос, «робо-ритм»/пинг-понг (нет min/max-clip-duration), рассинхрон на длинных.
Наши min-hold + cross-talk→wide + локальность уже били это по коду — но min-hold де-факто не
работал (потолок 1000), а cross-talk-порог не настраивался из UI. Оба преимущества
восстановлены. Существующий недеструктивный Enable/Disable-режим (`params.mode`) признан
достаточным — отдельная фича «хранение плана для ре-свитча» отложена.

TDD: `multicam-plan` +2 (>1000 коротышей, покрытие границ), `deterministic-pipelines` +1
(проброс smoothingWindow, RED-проверен), полный набор **751/751**.

## Мультикам: вывод скрытых параметров в UI (11.07.2026, live 6_SYNCED)

Задача — «параметры есть в коде, но не в UI». Каждый проверен на реальной 80-мин секвенции
через CDP: **валидировался результат на таймлайне** (сегменты плана), не только «значение
доехало». Это отсекло декоративные ползунки.

**Tier 1 — реальный эффект, выведено 3 ползунка:**
- `frameSec` («Шаг анализа» 50–200мс): 0.05→0.20 дал 278→**374 сегм**, framesAnalyzed 50316→12579.
- `maxAllSpeakersSec` («Длина вставки» 0.5–4с): 4→0.5 дал 278→302 сегм (короче wide-мосты).
- `variationsSeed` («Вариант» #1–20): при jitter>0 меняет рисунок разброса границ.
- Все уже форвардились пайплайном; добавлены только UI+glue. Регресс-тесты RED-проверены
  (temp-break форварда → падают).

**Валидация вскрыла мёртвые/спящие параметры** (не стали делать no-op ползунки):
- `wideOnSilence`/`wideOnOverlap` — **мертвы**: fallthrough `if (activeMic < 0) return wide`
  в `micToVideoTrack` перекрывал флаги (доказано: false→всё равно wide). **Оживлены**: при
  false тишина/перебивка держат ПОСЛЕДНЕГО спикера (4-й арг `lastSpeakerTrack`, дефолт true
  не меняется). Чекбоксы «Общий на паузах / на перебивках».
- `snapWindowSec`/`frameOffsetSec` — **спали**: snap-to-onset/silence в мультикаме не
  запускался, некому было считать источники. **Оживлены**: `MulticamPlan.computeSnapSources`
  выводит onset-ы речи и интервалы тишины из тех же RMS-кадров; `multicamFromAudio` считает
  их (только при snapWindowSec>0) и прокидывает. Ползунки «Привязка к паузам» (0=выкл) +
  «Сдвиг привязки».

Live на 6_SYNCED: дефолт (snap=0) — источники НЕ считаются (0 onset/0 silence, поведение
как раньше); snap=0.3 + hold-last-speaker — **7332 onset + 6585 silence** доехали в план,
wide 18%→16%. Все 4 параметра доехали до `buildSwitchPlan` из UI, консоль без ошибок.
