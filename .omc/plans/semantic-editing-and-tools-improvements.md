# План улучшений: семантический монтаж, маркеры, различение cut_silences / jump_cuts

Дата: 2026-04-16
Статус: черновик рекомендаций (не для немедленной реализации)

## 1. Текущее состояние (на основе кода)

### 1.1 Чат и семантика
- `client/shared/prompts.js`: tiered prompt (TIER0 + TIER1 по intent), one-pass принцип, classifyIntent/classifyComplexity. Добротно структурирован.
- `client/shared/transcript-structure.js:610` — `analyzeForCutsWithLLM`: локальные детекторы (filler/intro/outro/artifact) + LLM-классификация остальных сегментов чанками по `ANALYSIS_CHUNK_SIZE`. Кэш 30 мин.
- `client/unified/panel.js:2185` — `execAnalyzeTranscriptForCuts`: метки `content` и `digression` попадают в `toKeep`, всё остальное — в `toRemove`.
- `client/shared/find-moments.js` — семантический поиск на TF-IDF (без эмбеддингов). Токенизация, стоп-слова RU/EN, бонус за фразу целиком.

### 1.2 Маркеры
- `client/shared/deterministic-pipelines.js:376` — `chapterize`: topics через LLM (если нет в кэше) → markers → фильтр `<15с` между главами → `maxChapters` равномерным прореживанием.
- Типы маркеров: только `chapter` и `comment`. Premiere поддерживает больше (Web Link, Speech и др.).
- Fallback: time-based chapters каждые 30/45/60с в зависимости от длительности.

### 1.3 Jump Cuts vs Cut Silences
- `cutSilences` (pipelines.js:249): gaps между сегментами + ffmpeg silencedetect с gating `effectiveThreshold >= thresholdUsed`, `minDuration=1.0`, `padding=0.15`, `MIN_CUT=0.15`.
- `jumpCuts` (pipelines.js:493): **те же** gaps + **те же** ffmpeg silences, **БЕЗ** gating по threshold, `maxPause=0.5`, `padding=0.08`, `MIN_CUT=0.15`.
- В UI (index2.html:306-354): cutSilences имеет параметры min/thresh/padding, jumpCuts — только maxPause.
- **По сути 85% кода дублируется**, различаются только дефолты параметров и отсутствие threshold у jumps.

## 2. Проблемы

### 2.1 Чат — качество семантического монтажа
1. **Классификация на уровне сегментов Whisper, а не абзацев.** Сегмент ≈ фраза 3-10 сек, но смысловая единица — абзац 15-60 сек. LLM видит отрезок вне контекста соседей → классифицирует «content» фразу, которая на самом деле — часть отступления.
2. **Нет инструмента «отранжируй абзацы по релевантности теме».** Для запроса «собери ролик про X» LLM вынужден читать `get_transcript_structure` целиком и держать ранжирование в голове — хрупкий путь.
3. **`find_moments` на TF-IDF не матчит синонимы.** «Про деньги» не найдёт «прибыль, выручка, финансы». Для запросов по смыслу это критично.
4. **`propose_transcript_cuts` принимает только `removeIntervals`.** Для сборки (keep-logic) LLM должен инвертировать список в голове → ошибки границ, пропуски концовок.
5. **`digression` не режется.** В panel.js:2185 `digression` попадает в `toKeep` вместе с `content`. Пользователь просит «убери воду» → отступления остаются.
6. **Нет метрики уверенности.** Все метки equal-weight. Агрессивность удаления не регулируется.

### 2.2 Маркеры — по смыслу
1. **Жёсткий фильтр `<15с` между главами.** Интервью с быстрыми сменами темы теряет главы; длинные монологи не получают нужных.
2. **Имена глав не валидируются.** LLM часто возвращает «Часть 2», «Продолжение», `time-based fallback` ставит «Часть N» — бесполезно.
3. **Нет отдельного инструмента для хайлайтов.** Кульминации, инсайты, яркие цитаты технически делаются через `propose_markers` с `type=comment`, но отдельного пайплайна нет — LLM сам решает что хайлайт, нестабильно.
4. **Не используется `endSec` маркера.** В chapterize собирается `endSec`, но PP API его не применяет визуально.

### 2.3 Jump cuts / Silences — не различаются
1. **Алгоритмически почти идентичны.** Пользователю непонятно, когда что использовать.
2. **jumpCuts не gating-ит по порогу громкости** → может цеплять фоновый шум как «тишину».
3. **jumpCuts режет до нуля** → речь становится механической, «без дыхания», эффект клиппированной нарезки.
4. **Нет опции «сжать тишину»** (пауза 5с → 1с) — полезно для презентаций.
5. **Нет `minSegmentDuration`** — после резки могут остаться микро-клипы <0.3с, которые визуально мусор.

## 3. Рекомендации

### 3.1 Чат и семантика (приоритет: высокий)

**R1. `keepIntervals` в `propose_transcript_cuts`.**
Принимать альтернативу: список абзацев/интервалов, которые ОСТАВИТЬ. Бэкенд инвертирует в removeIntervals. Для «собери ролик про X» LLM выбирает top-N абзацев, не считая дырки в голове.
Acceptance: `propose_transcript_cuts({ keepIntervals: [...] })` работает; removeIntervals вычисляется автоматически; границы совпадают с паузами между абзацами.

**R2. Новый инструмент `rank_paragraphs_by_topic(sequenceKey, topic)`.**
Делает 1 LLM-вызов по всему транскрипту, возвращает массив `[{ paragraphIndex, relevance: 0-10, reason }]`. Агент берёт top-N по relevance × длительность ≤ target.
Acceptance: возвращает >=5 результатов для не-пустого транскрипта; sum relevance по топ-3 > sum relevance по нижним 3.

**R3. Перенос `analyze_transcript_for_cuts` с сегментов на абзацы.**
LLM классифицирует целые абзацы (есть контекст → больше точности). Для precise-границ реза используются sub-paragraph сегменты.
Acceptance: число LLM-вызовов падает в 3-5 раз (абзацев меньше сегментов); точность классификации растёт (оценивать вручную на 10 видео).

**R4. Фикс: `digression` → в toRemove по умолчанию.**
Или параметр `aggressiveness: 'gentle' | 'normal' | 'aggressive'`:
- gentle: удаляет только filler/artifact
- normal: + intro/outro/outtake/repeat (текущее)
- aggressive: + digression

**R5. Метка `weak_content`.** Добавить в классификатор. Позволяет тонкое управление («убери только явный мусор» vs «ужми до ключевого»).

**R6. Семантический `find_moments` через эмбеддинги.**
Если Cloud.ru предоставляет embeddings endpoint — кэшировать эмбеддинги абзацев (хранить в transcript-entry), запрос тоже эмбеддинг, косинусное сходство. Fallback на текущий TF-IDF если эмбеддингов нет.
Acceptance: запрос «деньги» находит абзац с «выручка/прибыль»; TF-IDF бы не нашёл.

### 3.2 Маркеры

**R7. Адаптивный минимальный интервал между главами.**
< 3 мин → 10с; 3-10 мин → 20с; > 10 мин → 45с. Вместо хардкода 15.

**R8. Отдельный пайплайн `/highlights`.**
Детектирует: ответ на прямой вопрос в тексте, числа/факты, восклицания, смены эмоции (если есть данные prosody). 1 LLM-вызов. Результат — маркеры `type=comment` с цветом (если Premiere API позволяет).
Acceptance: на тестовом видео 10 мин находит 3-7 highlights; каждый содержит цитату в `comment`.

**R9. Валидация имён глав.**
Если LLM возвращает «Часть N», «Продолжение», «Следующая часть» — заменить на первые 4 слова первой фразы главы. Если первая фраза — филлер, брать вторую.

**R10. Использовать существующие типы маркеров.**
`chapter` — структура, `comment` — highlights. Можно добавить `web_link` для ссылок на тайм-коды (если автор упомянул URL).

### 3.3 Jump cuts vs Cut silences — концептуальное разделение

**R11. Переосмыслить UX.**
- **«Убрать тишины» = гигиена**: убирает явно пустые паузы (≥1с), сохраняет естественный ритм речи.
- **«Jump cuts» = ритм**: агрессивно сжимает все паузы, YouTube-стиль, динамика.

UI-подписи в index2.html:307, 345 должны отражать это различие.

**R12. Единый детектор тишин — shared helper.**
```
detectSilenceIntervals(entry, { thresholdDb, minDuration, padding, source: 'gaps+ffmpeg' | 'gaps' | 'ffmpeg' }) → intervals
```
Обе функции зовут один helper. Сейчас 90% кода cutSilences и jumpCuts дублируется.

**R13. `jumpCuts` должен gating-ить ffmpeg silences по threshold** — как cutSilences. Сейчас не делает → ложные срабатывания на шуме.

**R14. `jumpCuts` — параметр `keepBreathing` (0/0.05/0.1с).**
Не резать до нуля, оставлять дыхание. Психоакустика: 50 мс естественной паузы vs 0 мс даёт разительно менее «клиппированный» результат. Дефолт 0.05с.

**R15. `jumpCuts` — параметр `minSegmentDuration` (0.3с).**
Если после резки сегмент длится <0.3с — объединить с соседним интервалом (поглотить как часть паузы).

**R16. `cutSilences` — опция «сжать, не вырезать» (`mode: 'cut' | 'compress'`).**
В режиме compress: пауза 5с → 1с. Для презентаций, обучающих видео, подкастов.

### 3.4 Общие улучшения

**R17. В proposal-area визуализировать интервалы.**
Сейчас `removeSummary` собирается, но в UI показывается только строка «Найдено N пауз». Добавить раскрываемый список с `[startSec–endSec] reason`.

**R18. Undo для монтажа.**
Сейчас есть Undo маркеров. Для razor-cuts добавить сохранение snapshot до apply и кнопку «Откатить последний монтаж».

**R19. `preview: true` в параметрах пайплайна.**
Явно в API — возвращает proposal без подготовки к mutation. Сейчас de-facto так и работает, но неявно.

## 5. Acceptance Criteria (общие)

- [ ] На запрос «собери ролик 45с про X» агент использует `keepIntervals` и делает один `propose_transcript_cuts` (R1, R2)
- [ ] «Убери воду» удаляет digression (R4)
- [ ] Jump cuts и Cut Silences визуально дают разный результат на одном материале (R11, R14)
- [ ] jumpCuts с `keepBreathing=0.05` звучит естественнее, чем до нуля (R14 — субъективная оценка)
- [ ] Главы для 3-мин видео ставятся каждые ≥10с, для 15-мин — ≥45с (R7)
- [ ] Имена глав никогда не «Часть N» и «Продолжение» (R9)

## 6. Риски

- **R2/R3 (перенос на абзацы):** если `buildParagraphs` разбивает плохо (длинные монологи без пауз), классификация абзацев будет грубой. Митигация: fallback на segment-level при paragraph.length < 3.
- **R6 (эмбеддинги):** зависит от доступности API. Митигация: feature-flag, TF-IDF как fallback.
- **R12 (shared helper):** рефакторинг существующего рабочего кода. Митигация: прогнать все тесты `tests/unit/*.test.js`, сравнить результаты до/после на 3-5 тестовых транскриптах.
- **R14 (keepBreathing):** может удлинить результирующее видео на 5-10%. Митигация: параметр в UI, дефолт 0.05с.

## 7. Что НЕ делать

- Не переходить целиком на эмбеддинги без fallback — зависимость от внешнего API.
- Не делать мега-пайплайн «автомонтаж всего»; пользователь хочет контролировать, что делает каждая кнопка.
- Не плодить настроек. Максимум 3 слайдера на карточку инструмента.
