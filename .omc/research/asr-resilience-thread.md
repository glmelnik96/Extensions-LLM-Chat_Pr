**Статус:** открыт, отложен 2026-05-31.
**Контекст:** возник в Phase 2 multicam, в ответвлении «Сначала ASR» от основной цепочки задач. Не доведён до решения (пользователь переключился на multicam-feature). Здесь — восстановленный контекст, чтобы можно было вернуться позже без повторного открывания.

---

## Проблема, которую решаем

Текущий ASR-pipeline ходит в **Cloud.ru Foundation Models / Whisper Large v3** (OpenAI-совместимый эндпоинт). Это «золотой» путь, но:

1. **Облако может быть недоступно** (сеть, токены, лимиты, плановые работы Cloud.ru) — пользователь получает hard-fail на длинных видео.
2. **На Windows возможны коллизии путей** при обращении к временным WAV/чанкам (см. отмеченный risk в Phase 1: пробелы / кириллица в пользовательских папках, как в `C:\Users\Глеб\...`).
3. **Локальный fallback на whisper.cpp** уже частично исследован, но не интегрирован в production-путь. Бинарь whisper.cpp есть, модели тоже — нет адаптера и переключения.

Цель: сделать транскрипцию устойчивой к падению облака **без потери качества по умолчанию** (cloud-first) и корректно работающей на Windows-путях.

## Дизайн-пространство (три направления, обсудить)

### A. Прозрачный auto-fallback (cloud → local)
- Единый клиент `transcribe(input, opts)` пробует Cloud.ru.
- На любой class-of-failures (network, 5xx, токен-fail, timeout > N) — автоматически идёт в whisper.cpp.
- UX: пользователь не выбирает, всё под капотом, в логе видна причина деградации.
- Плюс: одна точка контроля, никакой когнитивной нагрузки.
- Минус: непредсказуемость качества (local model меньше cloud-Large-v3), скрытая деградация без явного сигнала.

### B. Явный режим (cloud-only / local-only / auto)
- Setting в UI: «Cloud only / Local only / Auto (cloud first)». По умолчанию Auto.
- Каждый режим — отдельная стратегия, без неявной магии.
- Плюс: предсказуемо, отлаживаемо, видно где какое качество.
- Минус: пользователю надо понимать разницу, +1 setting.

### C. Pre-flight detection (single decision per session)
- В начале сессии: ping Cloud.ru + check whisper.cpp бинарь/модели.
- Решение принимается один раз, держится до явного refresh.
- Плюс: нет неожиданных переключений на середине задачи, чисто observable.
- Минус: «застрял» в local после восстановления cloud до конца сессии.

Лидер по компромиссу — **A с явной видимой деградацией в карточке/логе** (по сути A + честный сигнал из B).

## Windows-paths sub-tread

Отдельная подзадача (актуальна для всех трёх вариантов):
- `path.join` → forward-slash нормализация (как уже сделано в snapshot для `mediaPath`).
- Cyrillic-safe временные файлы: `os.tmpdir()` + UUID-suffix, никаких user-display строк в путях.
- ffmpeg + whisper.cpp оба читают UTF-8 пути нормально на современных Windows, но запуск процесса через CEP — проверить шеллинг (двойные кавычки + не звать через `cmd /c`, идти напрямую через `child_process.spawn` с массивом argv).
- Уже есть памятка в memory: SSH/Git Cyrillic workaround через `C:\ssh-home\.ssh` — там же отметить, что для CEP-spawn используем argv-массив, не shell-строки.

## Открытые вопросы пользователю (на момент возврата)

1. Какой вариант (A / B / C / гибрид) выбираем?
2. Что считаем «class-of-failures» для авто-переключения (network / 5xx / quota / timeout)? Какой `timeout > N` — 30 сек?
3. whisper.cpp модель по умолчанию — `large-v3` (медленно, точно), `medium`, или `small`? Trade-off скорости/качества зависит от типичной длины видео пользователя.
4. Нужен ли «no-local-fallback» режим для корпоративных тестов (где локальная транскрипция выключается принудительно)?
5. Куда сигналим о деградации — toast / отдельная строка в proposal-summary / лог-панель?
