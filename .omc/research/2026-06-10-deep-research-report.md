## 1. ЧТО НЕОБХОДИМО ПОПРАВИТЬ (подтверждённые дефекты)

### P0-1. Кэши анализа не инвалидируются после ripple-удалений
**Где:** panel.js:2796/2811 (ключ кэша = `matchedKey|tasks[|aggressiveness]`, без версии правок), TTL 30 мин (panel.js:2721).
**Факт:** ни одна из 6 точек вызова `ContextStore.applyRippleDeletionsToTranscript` (panel.js:1149, 2085, 2295, 2625, 3176, 4762) не сбрасывает `_labelsCache`/`_analysisCache`/`_analysisInFlight`.
**Последствие:** пользователь применил вырезки → жмёт «Убрать паразиты» повторно → получает метки со СТАРЫМИ таймкодами → ножи режут не там. Это прямое попадание в жалобу «инструменты иногда не работают».
**Фикс:** функция `invalidateAnalysisCachesFor(seqKey)` — вызывать во всех 6 точках. Либо включить `entry.editHistory.length` в ключ кэша.

### P0-2. Светодиод/карточки «Инструментов» блокируют audio-only инструменты
**Где:** panel.js:4552-4582 (`toolsRefreshLed`, `toolsUpdateCards`).
**Факт:** зелёный — только при `entry.segments.length`; иначе красный и ВСЕ `.needs-transcript` карточки выключены. Но «Тишина» и «Мультикам» работают по аудиоанализу и транскрипт им не нужен.
**Последствие:** пользователь без транскрибации не может запустить инструменты, которым она не требуется.
**Фикс:** третье состояние (синий/жёлтый «только аудио»), карточки делить на `needs-transcript` и `needs-audio`.

### P1-1. «Порог тишины» в UI — на самом деле дельта от громкости, а не абсолютный dB
**Где:** deterministic-pipelines.js:341-344 — `effectiveThreshold = userDelta > 0 ? floor(inputI - userDelta) : thresholdUsed`.
**Факт:** подпись в UI обещает абсолютный порог (дБ), код трактует значение как «на N дБ тише средней громкости».
**Фикс:** переименовать подпись («тише речи на N дБ») + tooltip; или честный абсолютный режим. Конкуренты (Recut, AutoCut, FireCut) дают **Auto-threshold** из аудиоанализа — у нас inputI уже есть, авто-режим почти бесплатен.

### P1-2. Перекрывающиеся removeIntervals: host и транскрипт расходятся
**Где:** tool-validators.js:144-178 (`validateTranscriptCuts` не проверяет пересечения/сортировку); premiere.jsx:1590 (применяет справа-налево БЕЗ слияния); context-store.js:376-384 (транскрипт-кэш перекрытия СЛИВАЕТ).
**Факт:** при overlap второй ripple на хосте вырезает уже сдвинутый материал → таймлайн теряет лишнее, а транскрипт-кэш считает иначе → все последующие операции по транскрипту бьют мимо. Overlap может даже создаваться нашим же `snapIntervalsToSegmentBoundaries` (две близкие вырезки прилипают к одной границе сегмента).
**Фикс:** merge-перекрытий в одном месте — в `execProposeTranscriptCuts` после snap'а (panel.js:2414) и в `execApplyTranscriptCuts` (panel.js:3156). Валидатору — error при пересечении после merge.

### P1-3. GLM-5.1 c thinking: 46–48 с до первого токена НА КАЖДЫЙ шаг агентного цикла
**Факт (live-замер 10.06.2026):** thinking=ON — 46–48 с TTFT; thinking=OFF — 0.3–0.8 с; gpt-oss-120b / DeepSeek-V4-Pro — ~0.4 с.
**Сделано:** ETA-индикатор («модель думает… Nс · обычно ~Mс», EMA по модели, ключ учитывает `#think`) — agent-loop.js + panel.js, 392/392 тестов.
**Осталось решить:** диалог из 3 шагов цикла = ~2.5 минуты раздумий. Варианты: (а) thinking=OFF для чата по умолчанию (качество tool-calling GLM-5.1 без thinking — проверить на эвалах); (б) thinking только на первом шаге; (в) роутить шаги с tool-calls на DeepSeek-V4-Pro.

### P2-1. 413 в чате: честная ошибка, но без авто-восстановления
**Где:** cloudru-client.js:351.
**Фикс:** при 413 — автоматически выкинуть старейшие сообщения истории (кроме system + последнего user) и повторить один раз; в ошибке оставить текущий совет.

### P2-2. Multicam: задокументированные ловушки без защиты
- **Bleed петличек:** `bleedMarginDb` по умолчанию может не спасать при громком прострое; нет диагностики «два микрофона почти одинаковой громкости 80% времени» (предупреждение пользователю).
- **Несколько клипов на дорожке:** молчаливый skip лишних клипов — нужно явное предупреждение в ответе инструмента.
- **maxHold wide-bridge:** поведение «мостим широкий план» не отражено в UI-подсказке.

### P2-3. apply_transcript_cuts можно вызвать без proposal
**Где:** panel.js:654 — защита только на уровне промпта («используй только если пользователь явно попросил»). Кодового гейта нет.
**Фикс (опционально):** требовать `confirm:true` в аргументах, если нет `_pendingProposal`, и возвращать validationError с инструкцией — мягкий кодовый гейт, не ломающий сценарий «без подтверждения».

### Опровергнуто аудитами (НЕ чинить)
- «paragraphs протухают после ripple» — ленивый rebuild уже есть: `isParagraphsStale` + `buildStructure` во всех 3 точках чтения (panel.js:1019, 2690, 2779); дрейф <1 с — осознанный допуск.
- «default filler-mode UI≠код» — код принимает 'normal' корректно (deterministic-pipelines.js:119-120).

## 2. ЧТО МОЖНО УЛУЧШИТЬ (проверенные shipped-решения конкурентов)

Полные таблицы параметров и источники — в `2026-06-10-competitors-report.md`. Самое применимое к нам:

### 2.1 Асимметричный padding с жёсткими полами (Timebolt)
Слева 0.09 с (мин 0.01), **справа ≥0.15 с — никогда меньше** (хвостовые сибилянты S/P). У нас `paddingSec=0.3` симметричный (panel.js:2410). Сделать `padLeftSec`/`padRightSec` с клампом правого ≥0.15. Это главный фактор «не звучит обрезанно» у всех конкурентов.

### 2.2 Review-список ↔ таймлайн (FireCut Advanced, канонический паттерн для CEP)
Список найденных тишин/паразитов в панели; клик по элементу = playhead к месту на таймлайне; крестик = исключить; per-item padding; потом одна кнопка «Применить». У нас уже есть proposal-карточка — добавить клик-навигацию по интервалам и исключение отдельных интервалов.

### 2.3 Markers-only режим (FireCut, AutoCut «detect only»)
У каждой деструктивной фичи — недеструктивный режим «просто расставь маркеры, где я бы резал». Дёшево (host уже умеет маркеры), строит доверие, обходит лимиты производительности ExtendScript.

### 2.4 «Avoid harsh cuts» guard (Descript)
Перед удалением паразита проверить зазор до соседних слов; **пропустить** удаления, которые обрежут соседнее слово (и сказать об этом). «Отказ от рискованного реза» — самый дешёвый прирост точности filler-removal.

### 2.5 Auto-threshold (Recut «Auto», AutoCut AI noise level)
Кнопка «Авто» у порога тишины — у нас inputI/lra уже измеряются, значение = `inputI - 25` как старт. Убирает ошибку №1 пользователя (неверный dB-пол).

### 2.6 J-cut / Constant Power на швах (FireCut J-cut offset, AutoCut transitions)
Опция «аудиопереход на каждом резе»: None (деф.) / J-cut N кадров / Constant Power. Проверенное решение слышимых швов именно в Premiere.

### 2.7 Поведение порога: Loose↔Tight один слайдер (FireCut Basic)
Один слайдер «плотность монтажа», внутри маппится на threshold+minDur+padding; Advanced — реальные параметры. У нас уже есть aggressiveness для LLM-анализа; ту же идиому распространить на «Тишину».

### 2.8 Удаление дублей-дублей (FireCut Remove Repetition)
Fuzzy-матчинг **только последовательных** повторов фраз по транскрипту (tolerance 0–80%, min phrase size, search radius), список тейков → tick «оставить этот». Никто не обещает глобальный поиск ретейков — честный скоуп. Кандидат на новый инструмент (у нас транскрипт уже есть).

### 2.9 FastForward вместо вырезания (Timebolt)
Альтернатива delete для тишин >10 с: speed-ramp. В Premiere API смена скорости недоступна (set_clip_speed отбит валидатором) — НЕ брать. Вместо этого: «Remove but keep spaces» (AutoCut) — вырезать аудио, сохранив тайминг, у нас реализуемо через mute/gain.

### 2.10 ETA/прогресс UX (индустриальный консенсус)
- Таймер с нуля + честные названия стадий («Транскрибация → Анализ пауз → Сборка плана»), никаких фейковых процентов.
- Диапазоны вместо точных оценок («обычно 30–60 с»), консервативно.
- Эскалация: 15–30 с — предложение «продолжить в фоне»; 60 с+ — фон по умолчанию.
- ✅ Уже реализовано: elapsed-таймер + «обычно ~Nс» из телеметрии (EMA). Дальше: стадии для длинных пайплайнов инструментов.

## 3. Multicam (глубокое исследование)
Полный отчёт с источниками: `2026-06-10-multicam-report.md` (AutoPod, Resolve 20 SmartSwitch — официальный PDF, AutoCut, Wraith, Descript, Eddie AI, Roland VR-50HD, LiveCUT, Shure IntelliMix, Stanford SIGGRAPH 2017, Barry Salt).

### Главный вывод
Наша архитектура (per-mic RMS + bleedMargin + min/max hold + wide на overlap/silence + snap-to-onset + razor/disable) **почти точно совпадает с shipped state-of-the-art**. Premiere и FCP нативного авто-multicam НЕ имеют — конкурируем только с плагинами. AutoPod (стандарт индустрии) — тоже volume-based, его метод Enable/Disable идентичен нашему. Жалоба №1 на AutoPod: «режет на чих/смех» — у нас та же уязвимость.

### Подтверждённые shipped-механики, которых у нас нет (по приоритету)
1. **Onset-confirm attack ~300–400 мс** (Shure gate hold 300–400 мс, LiveCUT Attack, OBS peak-over-window): новый лидер должен удерживаться 300–400 мс до коммита переключения. Прямо убивает ложный кат на чих/смех — худший failure mode всей категории. Атаку сделать асимметричной: медленно переключаться К новому, ещё медленнее отпускать.
2. **Edit Change Delay ~0.3 с ПОСЛЕ onset** (Resolve 20 SmartSwitch, официальный default из туториалов; Min Edit Duration 1.5 с = наш minHold): видеокат отстаёт от голоса → естественный J-cut. Подтверждено киноданными: Stanford SIGGRAPH 2017, α=0.9/β=0.1 — в реальных фильмах кат на ~90% паузы, прямо перед новым спикером. Мы сейчас снэпим ровно НА onset — добавить `cutDelaySec` (или −10% паузы).
3. **Reaction shots на длинном монологе** (AutoCut «Reaction Shots» toggle, Descript cutaways, VVD fuzzy rule): по истечении maxHold — иногда не wide, а 2–4 с крупного плана слушателя, вставка на границе предложения/паузы (Salt 2011: «реакции следуют длине предложений»).
4. **Именованные пресеты темпа** (AutoCut: Calm/Paced/Energetic/Hyperactive → minHold/maxHold/wideFreq/variation). У нас слайдеры Phase 2B уже есть — обернуть в пресеты.
5. **Wide-политика**: Resolve — «wide для intro/outro» (checkbox) и Wide Angle Frequency Low/Med/High; Roland Silent=OFF — альтернатива «держать последний план в коротких тишинах» вместо безусловного wide.
6. **Speaker priority** (AutoCut Normal/High/Very High): гандикап −2/−4 дБ не-приоритетным или длиннее hold ведущему.
7. **Group/Mix target на overlap** (Roland VR-50HD: параметр Mix): если существует 2-shot ровно с перекрикивающимися спикерами — предпочесть его полному wide.
8. **Post-cut re-arm** (Roland Time 0–30 с): lockout детектора после переключения — подавляет РЕШЕНИЯ, не только каты (smoothing не копит отложенный switch).
9. **Per-mic автокалибровка** (LiveCUT Automatic Level Adjustment, VVD relative differences): пол тишины и уровень речи per-mic из калибровочного прохода вместо глобального порога — сохраняет смысл нашего 6 дБ margin на разнокалиберных микрофонах.

### Что НЕ делать (проверено)
- **Диаризация (pyannote/whisperX) — никто не шипит её для multicam.** С раздельными дорожками атрибуция и так 100%; bleed решается относительным сравнением + VAD + (у Resolve) губами, не кластеризацией.
- Не обещать «умное» переключение сверх честного скоупа: даже Resolve, единственный AV-инструмент (губы, wide/closeup-классификация), ловит жалобы на медленный анализ.
